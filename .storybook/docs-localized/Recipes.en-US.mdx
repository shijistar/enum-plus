# Recipes

This page gathers the practical patterns that used to be spread across the long README: UI generation, localization, metadata, naming conflicts, extension, and recommended usage habits.

## Generate UI data from one enum definition

Use the core API when you only need plain structures:

```ts
StatusEnum.toList();
StatusEnum.toMap();
```

Use plugins when you want framework-specific output:

```ts
import { Enum } from 'enum-plus';
import antdPlugin from '@enum-plus/plugin-antd';

Enum.install(antdPlugin);

StatusEnum.toSelect();
StatusEnum.toMenu();
StatusEnum.toFilter();
StatusEnum.toValueMap();
```

## Localize labels instead of duplicating display maps

There are two common paths:

1. **Framework-agnostic**: use `localize`, `labelPrefix`, and `autoLabel` for plain string output.
2. **Framework-aware**: use integration plugins such as `@enum-plus/plugin-react`, `@enum-plus/plugin-i18next`, or `@enum-plus/plugin-vue-i18n`.

```ts
const StatusEnum = Enum(
  {
    Draft: { value: 'draft', label: 'draft' },
    Review: { value: 'review', label: 'review' },
  },
  {
    name: 'status.enumName',
    labelPrefix: 'status.',
    autoLabel: true,
    localize: (key) => dictionary[key] ?? key,
  },
);
```

## Use metadata as a lightweight static configuration system

Metadata is one of the biggest differences from native enums.

```ts
const WorkflowEnum = Enum({
  Draft: { value: 'draft', label: 'Draft', color: 'default', permission: 'edit' },
  Published: { value: 'published', label: 'Published', color: 'success', permission: 'read' },
});

WorkflowEnum.meta.color;
WorkflowEnum.findBy('permission', 'read');
```

This keeps display logic and business metadata close to the enum definition instead of scattering separate lookup tables around the app.

## Handle naming conflicts without giving up business names

When enum item keys overlap with built-in API names, enum-plus preserves business fields and exposes the built-in capabilities through symbol aliases such as `ITEMS`, `KEYS`, and `VALUES`.

Live demo: [Core / Naming Conflicts](?path=/story/core-query-and-transform-api--naming-conflicts)

## Compose enums and add shared methods

Use `raw()` to reconstruct definitions, and `Enum.extends()` to add shared helpers:

```ts
Enum.extends({
  toBadgeMap() {
    return this.toMap({
      keySelector: 'value',
      valueSelector: (item) => ({ label: item.label, color: item.raw.color }),
    });
  },
});
```

Live demo: [Core / Composition and Extension](?path=/story/core-localization-composition-and-extension--composition-and-extension)

## Recommended habits

- Name enum types with `PascalCase` and a clear business meaning.
- Prefer the standard shape for business-facing enums.
- Keep labels and metadata near the enum instead of maintaining separate maps.
- Decide the localization strategy early if labels are user-facing.
- Use plugins only where needed so the core package stays lightweight.
- Document important enum items with JSDoc in source code to improve IDE guidance.
